# image2-design-director

给具备图像生成能力的 Agent 加上一层“设计导演判断层”，让结果更接近真实用户会采用的图像成品，而不只是“能出图”。

[Skill 入口](./SKILL.md) · [文档导航](./docs/README.md) · [目标架构](./docs/target-skill-architecture.md) · [运行时记忆](./references/runtime-memory.md) · [发布检查](./docs/release-readiness-review-2026-04-23.md)

图像生成 Agent 原本常见的问题不是不会写提示词，而是缺少一层更稳定的判断：什么时候该直接生成，什么时候该先补关键信息，什么时候该只做最小修正，什么时候其实是前面对需求理解错了。`image2-design-director` 补上的就是这层分阶段判断、质量把关和经验复利机制。

兼容思路上，这个仓库面向所有支持 `SKILL.md` 工作流、且具备图像生成能力的 Agent 运行环境。它不绑定单一宿主，仓库规则层和本地运行时经验层也被明确分开。

## 它适合什么

这个 skill 的目标不是只覆盖海报、UI 或品牌图，而是作为一个通用的图像生成导演层，放在几乎所有图片生成请求前面。

它可以用于：

- 海报、宣传图、品牌图、公开发布图
- 社交媒体配图、封面图、信息流创意图
- 产品图、商品图、设备主视觉、产品样机图
- UI、UX、应用界面、数据看板、设计系统视觉图
- 插画、场景图、概念图、引导页视觉图
- 带参考图改造、带尺寸限制、多候选抽卡的图片任务
- 已有图像不符合预期后的定向修正

更直接一点说：

- 只要用户是在“要一张图”
- 且希望这张图更符合预期、更像可用结果

这个 skill 就应该优先介入。

## Agent 什么时候调用这个 Skill

如果一个 Agent 正准备做任何图像生成任务，默认就应该先调用 `$image2-design-director`。

尤其是下面这些情况，更应该调用：

- 用户给了参考图，希望做成类似的形式、风格、构图或气质
- 用户给了比例、尺寸、导出规格或多尺寸要求
- 用户希望一次出多张图来对比、抽卡或做风格测试
- 用户明确说“不要太 AI 味”“更像真实商业产物”“更像设计团队会交付的东西”
- 用户对结果要求是“完整可用成品”，而不是随便一张图
- 用户已经有上一版，但说“不是我要的”“方向错了”“语言不对”“不像成品”

即使是非常轻量、完全随手型的出图，这个 skill 也可以介入，只是会走更轻的判断和更短的路径。

## 我们支持什么

下面这张表是给用户看的，不要求先理解内部术语。

| 能力 | 现在支持什么 |
|---|---|
| 参考图适配 | 用户可以给参考图，我们会吸收它的构图、风格、镜头关系、信息密度和视觉气质 |
| 比例和尺寸控制 | 支持固定比例、固定像素尺寸、主视觉扩多个尺寸，以及尺寸适配时保护焦点和安全区 |
| 多张输出和抽卡 | 支持一次出多张候选图，用于风格测试、方向比较、抽卡选优 |
| 成品图和底图区分 | 能区分用户要的是直接可用成品，还是后续再排版的底图 |
| 文案和语言控制 | 支持指定中文、英文或其他语言，也支持限制图里只能出现哪些文字 |
| 文字添加与调整 | 支持在交付阶段补标题、副标题、说明文字、按钮文案，并做位置、层级和密度调整 |
| 二维码和固定元素处理 | 支持二维码替换、二维码后置、logo、角标、徽章等固定元素的落位和替换 |
| 已有结果修正 | 如果上一版方向不对、语言不对、资产类型不对，可以走修正流程而不是从头乱试 |
| 结构化提示词组装 | 会把模糊需求整理成更稳定的生成输入，而不是只做表面润色 |
| 设计质量判断 | 会判断这张图到底是不是用户真正要的那种图，而不是只看“漂不漂亮” |
| 交付阶段支持 | 支持从原始生成图继续推进到可叠字底图、可直接交付版本和多尺寸版本 |
| 经验沉淀 | 会记录高价值案例、失败原因和修正方式，让后续类似任务更稳 |

## 它是怎么工作的

这个 skill 的特色，不是“有一段更长的提示词”，而是它有一套完整的分阶段机制。

### 第 1 步，先确认你到底要什么图

先判断：

- 最终资产是什么
- 是要完整成品还是底图
- 默认文案语言是什么
- 图里允许出现哪些文字
- 什么结果才算“现在就能用”

这一步的目的，是防止系统一上来就开始生成，但做的根本不是你要的东西。

### 第 2 步，再决定这轮应该怎么跑

这里会在 4 种方式里选一种：

- `direct`
  - 需求已经很清楚，直接生成
- `brief-first`
  - 还差少量关键信息，先补 1 到 3 个高杠杆信息再生成
- `repair`
  - 方向大体对了，只需要做最小修正
- `contract_realign`
  - 前面对需求理解错了，要先重建任务理解，再继续生成

这些名字是内部标签，但对用户来说可以理解成：

- 直接做
- 先补关键需求
- 小修
- 先纠正理解再做

### 第 3 步，把需求装成稳定提示词

我们不会只堆很多风格词，而是会按固定顺序整理：

- 任务目标
- 主体和场景
- 结构和构图
- 文字层
- 风格层
- 不能违反的约束

这样做的原因是，稳定结果更依赖“结构正确”，而不只是“提示词写得长”。

### 第 4 步，生成后先验收，再决定下一步

生成完不会直接宣布结束，而是会判断：

- 它是不是用户要的资产类型
- 它是不是完整成品
- 语言和文字是不是对的
- 还有没有明显 AI 味或跑偏

如果结果没过线，就不会假装完成，而是进入修正。

### 第 5 步，如果进入交付阶段，再处理文字、二维码和尺寸

当任务不只是“出一张图”，而是要变成真正能发的资产时，这一层会继续处理：

- 标题、副标题、说明文字的添加与调整
- 二维码替换、二维码后置和扫码可用性
- logo、角标、徽章等固定元素的落位
- 一个主视觉扩成多个比例或尺寸版本

这一层的目标，是把“看起来不错的图”推进成“真的能交付的图”。

### 第 6 步，把经验留下来

每一轮高价值任务都会尽量留下：

- 原始需求
- 最终提示词
- 输出结果
- 为什么成功
- 为什么失败
- 下一轮该怎么改

这样这个 skill 不是每次都从零开始。

## 为什么它能在很多图片类型里更稳定

不是因为它预设了“某一种图”的最佳提示词，而是因为它用了几层更通用的稳定器：

- 先对齐交付物，再生成，减少“做错资产类型”
- 先区分成品和底图，减少“半成品误交付”
- 先锁定语言和文字范围，减少“文本乱跑”
- 先判断该直接做、补信息、小修还是重建理解，减少无效试错
- 生成后做资产级验收，而不是只看画面氛围
- 把失败原因和修正路径沉淀下来，形成经验复利

## 用户可以直接提的常见请求

- “我给你一张参考图，按这个构图和气质做一版，但主题换成我们的产品。”
- “保持这个风格，给我做成 `4:5`，同时再出一个 `1:1` 版本。”
- “给我 3 张不同方向，我想抽卡选一张最像品牌主视觉的。”
- “先给我一张可以直接发的完整海报，不要底图。”
- “我后面自己排字，你先给我一个干净的可叠字底图。”
- “这版方向对了，但不像成品，帮我做一次最小修正。”
- “这不是我要的资产类型，重新按仓库首页主视觉来理解，不要再做成海报。”
- “中文对话就用中文文案，项目名保留英文，其余不要乱出英文小字。”
- “这张图里只能出现项目名、主标题和副标题，不允许额外小字。”
- “这个主视觉后面还要扩 `16:9`、`4:5`、`1:1`，请按可扩展交付来做。”
- “这张海报的二维码要替换成新的，并重新调整右下角留白和位置。”
- “标题先保留，副标题和按钮文案帮我补上，并把层级调顺。”

## 典型输入信息

用户不需要一次把所有字段都说全，但下面这些信息会显著提高命中率：

- 参考图或旧版本
- 最终资产类型，例如“品牌宣传海报”“仓库首页主视觉”“信息流创意图”
- 是否要完整成品，还是只要底图
- 比例、尺寸或导出要求
- 需要几张候选图
- 固定文案、logo、二维码、角标或徽章
- 当前会话语言之外，是否要指定别的文案语言
- 什么结果才算“现在就能用”

## 它和普通“提示词增强器”的区别

- 它不是只把一句话扩写得更长，而是先判断任务该怎么推进
- 它关注的是“这是不是用户真正要的那张图”，不是只关注提示词表面工艺
- 它会区分直接生成、补需求、小修和重建理解，而不是一把梭地不断重试
- 它会区分成品、底图和交付阶段，而不是把所有输出都视为同一种结果
- 它把失败样本和纠偏规则当成一等公民，而不是只收集成功模板

## 快速开始

### 1. 安装到你的 skill 目录

目前这个仓库主要提供手动接入方式。最简单的是把整个仓库 clone 到你的 agent skills 目录：

```bash
git clone https://github.com/boyzcl/image2-design-director.git "$AGENT_SKILLS_HOME/image2-design-director"
```

如果你的运行环境没有统一的 `AGENT_SKILLS_HOME`，也可以直接把仓库放到该宿主实际读取 skill 的目录里，或把 `SKILL.md`、`references/`、`patterns/`、`scripts/` vendoring 到现有系统。

### 2. 先读核心入口

第一次使用，建议按这个顺序：

1. [SKILL.md](./SKILL.md)
2. [docs/README.md](./docs/README.md)
3. `references/` 下的核心协议文档

### 2.5 调用方式

显式调用时，直接使用：

```text
$image2-design-director
```

也支持自然语言直接触发，只要你的 Agent 宿主会读取 `SKILL.md` 并按技能描述进行路由。

### 3. 可选启用运行时记忆

如果你只想使用规则层，直接读 skill 即可。

如果你想启用本地经验沉淀，再配置运行时根目录，并使用脚本：

```bash
python scripts/init_runtime_memory.py --host generic
python scripts/read_runtime_context.py --host generic --raw-only
```

运行时也可以通过 `IMAGE2_DESIGN_DIRECTOR_RUNTIME_ROOT` 显式指定本地运行时根目录。详情见 [references/runtime-memory.md](./references/runtime-memory.md)。

## 当前边界

- 本地自动晋升只作用于本地运行时 / 本地 skill 参考层
