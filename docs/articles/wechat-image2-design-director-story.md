# 我为什么做了 `image2-design-director`，AI 生图真正缺的不是 prompt，而是交付闭环

事情是这样的。

前段时间 Image2，也就是很多人说的深图，刚出来的时候，我第一时间就去试了。

说真的，我当时是兴奋的。不是那种看完几张 demo 以后礼貌性地点点头的兴奋，而是真的有点被打动了。因为它直接出图的效果，确实很好。很多画面你一眼看过去，会立刻觉得，哦，这玩意已经不是以前那种一股子 AI 味、只能拿来发朋友圈感慨一下的阶段了，它是真的能把氛围、材质、构图、品牌感，一口气推到一个很高的水位。

我一开始的判断其实很朴素，我觉得这下图像生成应该又往前走了一大步，很多以前麻烦的事情，可能终于能被更顺地做掉了。

结果很快我就发现，不是这么回事。

它当然能出好图，而且不少时候出得还挺惊艳。但你只要把要求往真实交付那边推一步，问题就开始暴露了。你想精确控制一点东西，开始费劲。你想改字，开始费劲。你想让二维码真的能扫，基本别指望模型自己搞定。你想把 logo、badge、固定元素、价格牌、日期这种东西稳稳地放进去，难度一下就上来了。你想先拿一张底图，后面再做替换、再做多尺寸、再做交付适配，事情又会变得非常微妙。

也就是说，它不是不会出图。

它的问题是，离稳定、可控、可交付，仍然有一段很明显的距离。

这段距离，我后来反复想，越来越觉得它不该被理解成一个简单的 prompt 问题。

因为如果一个系统真正的问题只是 prompt 不够好，那你通常会看到另一种现象。你多试几轮，多改几次，多补一点描述，它会越来越接近正确答案。可我在 Image2 这类任务里遇到的很多失败，不是这种失败。它们经常是上游就错了。

资产类型做错了，本来要的是品牌发布图，最后给你做成概念板。

成品度做错了，本来要的是能直接交出去的成品，最后给你一张看起来还行、但明显还得继续折腾的中间稿。

语言策略做错了，本来应该只有一句主标题，它却自己长出一堆字。

表达机制也做错了，本来应该重建理解，或者应该走后置、走 hybrid、走确定性叠加，它却还在表面修补，还在赌下一轮模型会不会突然良心发现。

我慢慢意识到，这里面最缺的东西，不是一个更会扩写 prompt 的助手，而是一层更像设计导演的判断系统。

这也是我后来做 `image2-design-director` 的真正起点。

如果你看过我之前对 `AI native loop` 的一些思考，你应该能明白我为什么会往这个方向走。因为我越来越不相信一次 prompt 就能把复杂交付问题解决掉。我也越来越不相信，每次都从零开始碰运气，是一个长期能成立的工作流。

我更相信另一条线。

目标要有完整上下文，过程不能只靠一次 prompt，每一轮经验应该能积累，系统应该越用越稳，而不是每次重新试错。

这套想法放到代码、写作、产品设计里都成立，放到生图里其实也一样成立。只是图像生成这件事，过去很容易被惊艳感掩盖。因为模型第一次把图做漂亮的时候，我们太容易误以为事情已经成了。可一旦任务开始碰到精确文本、二维码、后置替换、固定元素、多尺寸适配、信息可信度这些现实问题，你就会发现，真正麻烦的部分，恰恰是那些 demo 最不愿意告诉你的部分。

所以我做 `image2-design-director`，不是因为我想再做一个 prompt skill。

恰恰相反，我是因为越来越确定，这件事如果继续按 prompt skill 的方式理解，天花板会非常低。

现在市面上跟它相邻的能力，我大概会分成几类来看。

第一类是 `prompt rewrite`。这类东西很有价值，Leonardo 的 `Improve a Prompt`、OpenAI 图像生成里的 `revised_prompt`、Adobe Firefly 的一些 prompt guidance，本质上都在做这件事。它们帮你把原始输入整理得更像模型能执行的话。这很好，尤其对新手很友好，但它解决的是模型可执行性，不是交付物正确性。它不会告诉你这轮到底该直接生成、先补 brief、做 repair，还是干脆重建理解。

第二类是 `style / character / composition reference`。Midjourney、Ideogram、Firefly、Krea 在这块都很强。你给参考图，它帮你保风格、保角色、保构图、保结构。这一层我也很喜欢，因为真实工作里确实大量存在这种需求。但这类能力大多默认一件事，只要像参考图，就算成功。可真实任务里，像，从来不是唯一标准。它还要看资产类型是不是对的，文字策略是不是对的，成品度是不是对的，后面还能不能继续交付。

第三类是 `brand / template / layout consistency`。Canva 的 Brand Kit、Magic Layers，Adobe Firefly 的一些风格和模板体系，Recraft 的品牌风格能力，都在往这边走。这比单张好图更接近真实生产，因为它开始碰品牌一致性、模板复用、多尺寸适配这些问题了。但很多时候，它仍然默认用户已经知道自己要什么，默认任务理解层是对的。可现实里，最贵的错误往往不是图做丑了，而是一开始把任务理解错了。

第四类是 `variation / batch extension`。这一类很适合同风格探索、同系列延展、多候选发散，我自己也很喜欢用。可它仍然有一个老问题，如果合同本身错了，批量变体只会把错误扩散得更漂亮。

第五类是 `editable delivery`。这类能力离真实交付最近，因为它承认 AI 输出不是终点，后面还要继续编辑、改尺寸、换元素、补文案。问题在于，很多产品把它做成编辑器功能，编辑是有了，但它不会自然回流到前面的任务判断里。也就是说，它能帮你修，却不一定能帮你提前判断这张图值不值得进入修的阶段。

所以后来我给 `image2-design-director` 找的位置，其实越来越清楚了。

它不是去替代这些能力。

它是站在这些能力上面，做一层更高的组织。

如果要用一句最短的话来讲，我现在会把它定义成一层 `设计导演判断层 + 可交付闭环层 + 经验复利层`。

这三层缺一不可。

第一层是判断层。也就是它先不急着出图，先判断你到底要交付什么资产。仓库里现在把这个东西写成了显式的 `asset contract`。里面至少要锁几件事，最终交付物类型是什么，是完整成品还是底图，默认语言是什么，允许出现哪些可读文字，版式到底交给模型、后处理还是混合路径，最后的验收线是什么。

你会发现，这跟普通 prompt skill 的起手式已经完全不一样了。普通 prompt skill 的默认动作是听完一句话就开始润色 prompt。`image2-design-director` 的默认动作是先把交付合同锁住。因为没有这层合同，你根本不知道自己是在做品牌发布图、README hero、信息对比图，还是一张等着后面继续加工的交付底图。

第二层是路由层。合同锁住以后，它不会立刻假装所有任务都适合模型直出，而是先问，图里有哪些东西会被读者当成事实，这些信息到底可不可信，应该由什么系统来承载。

这也是为什么仓库里后来单独长出了 `information reliability gate` 和 `representation strategy`。

比如你要做一张纯品牌氛围图，那就很可能是低事实负载，适合走 `model_direct_visual`。

如果你要的是有限标题、有限短文案的 feature visual，它可能还能走 `model_visual_with_limited_text`，但要把文字纪律收得很死。

如果你要放二维码、logo、日期、CTA、后面还得多尺寸复用，那就更像 `visual_base_plus_post`。

如果你要做一张高事实敏感的对比图，里面有精确价格、时间口径、对比结论，那就先走 `information reliability gate` 锁口径，再优先让 Image2 直出完整成品。只有二维码、Logo、Exact copy 或已锁定数值需要逐字逐数一致时，才进入局部后期替换。

这一层特别重要，因为它真正把一个以前经常被含糊处理的问题讲清楚了。

不是所有信息都该让模型一次性生成。

不是所有失败都该靠再来一版解决。

有些任务，错在 prompt。

有些任务，错在 representation。

有些任务，压根就不该让模型承担那一层信息。

而且走到这里，它还不会粗暴地只剩下一个生成按钮。现在这套 skill 已经把 route 显式收成了几条路径。合同清楚、gate 结果清楚，就走 `direct`。缺几个高杠杆字段，就走 `brief-first`。已有结果但主线没错，只是局部出问题，就走 `repair`。如果连资产类型、成品度、语言策略、metric 口径都理解错了，那就不是修一修 prompt，而是直接 `contract_realign`，先把理解重建，再决定下一轮怎么生成。

第三层是验收和交付层。这个也是我觉得很多 image skill 里最容易被省略的一环。以前大家很容易把验收理解成一个很模糊的问题，好不好看，像不像设计图，顺不顺眼。可对真实资产来说，这种验收太弱了。

所以现在这套 skill 里，评估至少会同时看 `asset type fidelity`、`information reliability`、`representation fit`、`delivery integrity`、`misleading risk` 这些东西。它不是只问画面漂不漂亮，而是问，这是不是对的资产，图里关键信息是不是可信，表达机制是不是匹配，后面还能不能继续交付，会不会误导人。

你如果继续往下做交付，比如叠标题、放 badge、加二维码、补日期、做多尺寸，它还要再过一层 `delivery viability gate`。

这个 gate 我特别看重，因为它把很多人平时凭经验拍脑袋做的事，收成了一个正式的 go / no-go 判断。不是看哪里空就往哪里塞字，而是先判断当前这张图还有没有结构能力继续承载信息。有没有撞到硬保护区，软保护区擦边风险多高，overlay footprint 会不会直接把版式压垮。如果答案是不行，正确动作不是继续硬排，而是回到上游重生成，或者干脆换 representation mode。

这也是我为什么一直觉得，二维码、改字、固定元素、后置替换这些问题，不能只被当成小工艺问题。

很多时候它们暴露出来的，其实是整条路径选错了。

而 `image2-design-director` 最让我在意的一点，还不是上面这些，而是最后一层，经验复利。

因为大多数方案优化的，其实都是单轮生成。

这一轮图更好了，结束。

下一轮再来，很多经验还是散在聊天记录里，靠人脑记，靠模糊印象记，靠团队里那个最熟的人记。

我一直觉得这很亏。

所以这个 skill 从一开始就把 `runtime capture` 当成主链路的一部分。每次生成或编辑之后，不只是记录做了什么图，还会尽量把这轮为什么这样判断、为什么走这条 route、为什么选这个 representation mode、为什么 overlay 被允许或者被拦下、最后失败属于 contract、reliability、representation 还是 viability，全部留下来。

仓库里现在已经把这条脊梁写得很明确了，capture 至少会回溯 `brief -> final_prompt -> image_prompt -> image path -> evaluation`，而且经验不是直接冲进长期规则里，它会先留在 runtime，再经过 review，最后才有机会被提升成 field note、pattern 或长期知识。

我很喜欢这一层。

因为它带来的是一种很少见的东西，系统会开始记住自己是怎么变稳的。

这不是聊天记录意义上的记住。

是工作协议意义上的记住。

也正因为这样，`image2-design-director` 在我眼里才不只是一个生图 skill，而更像一个围绕图像交付搭出来的 AI native loop。它不只负责把一张图做出来，它还负责让下一张图别再从零开始交学费。

如果这套东西还停在理念层，那它当然没什么好说的。问题是，它现在已经不是停在理念层了。

仓库里已经把 `asset contract`、`information reliability gate`、`representation strategy`、`delivery viability gate`、`outcome accountability` 这些东西全都写成了显式协议，而且不是只有文档，还有实际可跑的 benchmark、overlay checker、post-processing 路径和 runtime 记忆链路。

4 月 23 日那轮真实回归，我觉得很能说明问题。

低事实负载的品牌发布图场景里，候选方案从 77 分提到了 97.6 分，真正把单标题纪律和 complete asset 成品感收紧了。

README 配套的 limited-text visual，从 61 分直接提到 91.2 分，不再轻易滑向那种自己长出很多字的信息板。

高事实敏感的精确数据对比图，普通做法让模型自己写精确数字，结果只有 57.8 分，而且 `misleading_risk` 很高。换成 `模型主视觉 + 确定性 exact-value overlay` 以后，分数到了 85，正式过线。这件事对我来说很关键，因为它说明 representation layer 不再只是理念上对，是真的能把高风险任务从不可信里拽出来。

还有 overlay 这块，checker 也不是只存在于文档里。真实底图上已经跑出了 `overlay_allowed`、`overlay_allowed_with_limits`、`overlay_not_allowed_regenerate` 三种状态，而且和实际 overlay 结果是对齐的。也就是说，这套系统开始具备一种很重要的能力，它不只会继续做，它也会在该停的时候停。

当然，现在它还远远没到可以吹成完美系统的程度。

比如文档里也写得很坦白，中文标题的工艺还要继续观察，deterministic overlay 的排版质量还偏 MVP，当前 viability checker 还是 box-level，不是语义分割级。你如果真想做特别复杂的 dense info layout，它也不会 magically 变成万能设计师。

但我反而觉得，这些限制被清楚写出来，本身就是这套东西可信的一部分。

因为它说明这不是一个营销故事。

它知道自己已经解决了什么，也知道自己还没解决什么。

回到最开始我试 Image2 的那个兴奋点上，我现在的感受其实比那时候更清楚，也更冷静一点。

我依然觉得它很强，而且是非常强。直接生图的上限，真的已经高得很吓人了。

但我现在更确定，下一阶段真正重要的，不是让模型再多惊艳我们一次。

而是让图像生成这件事，从惊艳 demo，变成稳定工作流。

我们真正需要的，也不是一个更会出图的系统。

我们需要的是一个更稳定地产出真正可用图像资产的系统。

`image2-design-director` 就是我沿着这条线做出来的一个回答。

它不是另一个 prompt enhancer，不是把一句话润色得更华丽，不是给模型多塞几个风格词，也不是简单把几个工具拼在一起。

它想做的，是把 `asset contract -> route -> generation -> acceptance -> delivery -> runtime accumulation` 这条链真正接起来，让图像生成第一次更像一个能被长期经营、能被持续纠偏、能把经验留下来的生产系统。

如果只看单轮结果，这件事可能没有某一张特别炸裂的图那么抓眼。

可如果你已经开始真的拿 Agent 做图，已经开始碰到精确控制、改字、二维码、后置替换、可交付资产这些问题，你大概会明白，我为什么会觉得这条路值得认真做。

因为我们差的从来不是再多一次惊艳。

我们差的是，终于有一套东西，能把惊艳变成可持续交付。
